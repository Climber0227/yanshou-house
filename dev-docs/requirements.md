## 设计决策变更记录

| # | 原始需求 | 实际实现 | 原因 |
|---|---------|---------|------|
| 三.5 | 整改人可以是任何一个扫码加入小程序的人 | **查验员上报时通过姓名+手机号指定整改人** | 现场需要精准指派责任单位，而不是等人扫码认领，确保责任到人 |
| 三.4 | 提示推送（短信或小程序通知） | Mock 阶段先在 App 级做联网自动上传队列+通知列表，**外部推送等后端对接** | 短信/模板消息需要后端+微信服务端，纯前端无法完成 |
| 三.7 | 增加选择无问题的功能，每项勾选无问题此户完成验收 | 实现为"无问题"按钮快速标记 visual 验收，整户验收需全部问题闭环后上报人审核 | 分项验收和整户验收拆分为两步，避免误操作 |

## 一、小程序包含五大模块

| 模块名称 | 核心功能 |
|----------|----------|
| 1. 户码管理（基础数据） | 1. 楼栋/单元/层/户初始化：导入 2-11# 楼所有户的层级结构<br>2. 生成二维码：每户（如 2#楼西单元 1 层西户）生成唯一二维码，张贴于入户门处<br>3. 扫码直达：支持微信扫码，自动识别户信息并跳转至该户的问题上报页 |
| 2. 问题上报（现场查验） | 1. 多级选择：选择户内观感（墙面、地面、顶棚、门窗等）、实测（室内净高、净开间、洞口尺寸、平整度、垂直度等）、公区（观感、实测）。“问题描述”支持从预设列表中选择或自定义，如“东墙墙面空鼓”<br>2. 多媒体记录：强制拍照上传，可选录视频、语音备注<br>3. 人员标注：自动记录上报人/验收人 |
| 3. 整改流程（闭环管理） | 1. 自动/手动派单：问题上报后，系统自动推送通知至《方案》中对应的“整改责任单位”负责人及“整改盯控人”<br>2. 状态标记：状态分为“待整改”→“整改中”→“待复查”→“已闭环”<br>3. 过程记录：整改责任人可上传整改照片、填写整改措施<br>4. 复查验收：监理/建设单位人员收到复查通知，现场复核，合格后签署电子确认 |
| 4. 一户一档（资料生成） | 1. 动态看板：进入任意一户的二维码，可查看该户所有历史问题、整改状态、验收合格与否<br>2. 报告生成：按《附件1：观感表》、《附件2：实测表》、《附件3：公区表》格式，自动填充已录入数据，生成电子档案<br>3. 结果汇总：自动汇总生成《附件4：分户验收结果汇总表》，统计每户的初评/终评结论 |
| 5. 数据看板（管理统计） | 1. 宏观进度：实时显示“总户数”、“已排查户数”、“问题总数”、“已整改数”、“整改完成率”<br>2. 问题热力图：按“部位”或“问题描述”统计高频问题（如“墙面空鼓”的出现频次及分布楼层）<br>3. 整改进度：按“楼层”或“楼栋”统计各责任单位的整改进度，形成红黄绿灯预警 |

## 二、关键开发说明与技术建议

### 1. 数据架构与角色权限

**角色定义：**

| 角色 | 权限 |
|------|------|
| 管理员 | 拥有最高权限，可查看所有数据、配置人员、管理楼栋信息 |
| 查验员 | 仅可上传问题，并可对分配给自己负责区域的问题进行复查确认 |
| 整改员 | 仅能接收指派给自己的整改任务，上传整改证明 |
| 业主（未来） | 可扫码查看自己房屋的“一户一档”信息，但无修改权限 |

**数据权限：** 不同角色只能看到与自己责任区域相关的问题。

### 2. 核心逻辑与流程设计

**问题生命周期：**

```
上报（未整改）→ 派单 → 整改（整改中）→ 申请复查 → 复查确认 → 已闭环
```

**整改时限提醒：** 对超过 24 小时未复查或超出整改期限的任务进行自动提醒（短信或小程序通知）。

### 3. 与现有方案的兼容性

**表格重现：** 小程序的导出功能必须能完美导出与《附件 1-5》格式一致的 Excel 或 Word 文档，避免二次人工整理。

### 4. 技术选型与部署

- **平台：** 建议使用微信小程序，方便用户扫码即用，无需额外下载 APP
- **二维码生成：** 使用稳定的二维码生成接口，保证在户门处张贴的二维码清晰耐用，且支持图片防伪（如水印）

### 5. 要点

1. 首先确保 **“扫码 - 上传 - 整改 - 闭环”** 这条核心流程的稳定性和易用性，这是现场人员最在意的环节
2. 明确终端设备：由于现场环境复杂，要求系统在安卓和 iOS 手机上均能良好运行，且能适应不同程度的网络信号（需支持离线暂存、联网后自动上传功能）
3. 数据安全性：此类数据涉及未来业主的房屋质量信息，务必做好数据加密和权限控制，防止信息泄露
4. 交付物要求：除了小程序本身，还需要交付管理员后台（用于运营配置、数据导出、人员管理）以及系统开发文档

## 三、其他建议

1. 导出的二维码文字信息应该是全的（二维码下边应该显示楼号单元号房号等信息）

2. 设置总问题库、分楼号问题库和房间号的问题库，问题库需要自动关联一个一个上传的问题（问题都是单独上传的），并显示清楚房号，不需要再单独整理问题库，直接导出就可以使用

3. 问题整改完成后要有审核，相应房号内问题全部整改完成，需要问题上传人审核同意后验房结束，首页进度同步更新

4. 增加提示推送功能，设置对应的责任人，问题上传完成后需要推给相应的责任人，问题上传可以选择整改期限

5. 整改人可以是任何一个扫码加入小程序的人，谁上传的东西谁有修改权限

6. 应该有同类问题或者通用问题一键上传功能（例如：1002、1003、1004、1005 这 4 户都有地面开裂的问题，我上传问题和照片的时候就可以多选这几个房间）

7. 增加选择无问题的功能，每一项都勾选无问题此户完成验收
