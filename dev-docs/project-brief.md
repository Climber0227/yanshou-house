# Project Brief — 分户验收小程序

## 项目边界

- 项目名称：house-acceptance（分户验收小程序）
- 项目是什么：住宅工程质量分户验收的微信小程序，让现场查验人员扫码上报问题、整改人员接收任务、监理复查确认，形成"扫码→上报→整改→闭环"的完整链路。
- 项目不是什么：不是物业管理软件、不是施工管理系统、不是业主报修平台（业主功能是未来扩展）。
- 第一批用户：查验员（现场质量检查人员）、整改员（施工方整改负责人）、管理员（项目管理者）、监理/建设单位人员。
- 使用场景：住宅工程交付前的分户验收现场，查验员入户检查，扫二维码上报质量问题，整改责任人收到通知后进行修复，监理复查合格后闭环。
- 一句话定位：用微信小程序替代纸质验收表，让分户验收"扫码即用、拍照即证、整改可追、一档可查"。
- 交付对象：客户交付（正式上线微信小程序）

## 用户问题和价值

- 用户现在的痛点：
  - 纸质验收表填写繁琐、容易丢失、后期汇总工作量大
  - 问题整改跟踪困难，不知道哪个问题改了、哪个没改
  - 验收数据和照片分散在个人手机里，无法统一管理
  - 一户一档整理费时费力，容易出错
- 为什么值得做：分户验收是住宅交付的强制环节，目前大量项目仍用纸质流程，数字化改造直接提升效率、减少纠纷。
- 同类产品、参考项目或竞品：无明显竞品参考，以需求文档中附件表格格式为准。
- 可以学习的地方：微信小程序扫码能力、离线数据暂存方案、电子签名组件。
- 不能照抄的地方：验收表格格式（观感表、实测表、公区表）必须严格按需求文档附件格式输出，不能自己设计。
- 本项目要比参考方案更适合用户业务的地方：与现有附件表格格式完美兼容导出，避免二次人工整理。

## 第一闭环

用最小完整流程描述第一版必须跑通什么：

```text
查验员扫码进入某户
  -> 选择验收类型（观感/实测/公区）
  -> 选择/自定义问题描述，拍照上传
  -> 系统自动/手动派单给整改责任人
  -> 整改责任人收到通知，进行整改
  -> 上传整改照片和整改措施
  -> 申请复查
  -> 查验员/监理现场复查确认
  -> 合格后电子确认，闭环
  -> 该户数据进入"一户一档"
```

- 第一闭环输入：查验员扫码 → 选择问题类型 → 拍照 → 提交
- 第一闭环输出：整改任务通知 → 整改证明 → 复查确认 → 状态变为"已闭环" → 数据归档
- 用户可见成功标准：一个问题的状态从"待整改"走完"整改中→待复查→已闭环"，在"一户一档"中可查看完整过程记录。
- 失败时用户会看到什么：网络异常时有离线暂存提示；必填项缺失时有表单校验提示；权限不足时有明确提示。

## MVP 范围

- 第一版必须有：
  1. 户码管理：楼栋/单元/层/户数据导入、二维码生成与展示（二维码下方显示楼号-单元-房号）
  2. 问题上报：多级分类选择（观感/实测/公区）、拍照上传、问题描述（预设+自定义）、人员自动记录
  3. 同类问题批量上传：多户复选同一问题描述和照片，一次提交
  4. 无问题标记：每项验收可勾选"无问题"快速通过该户
  5. 整改期限设置：上报时可选整改期限，超时自动提醒
  6. 整改流程：自动派单通知、状态流转（待整改→整改中→待复查→已闭环）、整改照片/措施上传、复查电子确认
  7. 问题库自动关联：上传的问题自动归集到对应楼号和房号的问题库，导出即可用，无需人工整理
  8. 管理后台：楼栋数据管理、人员管理、角色权限配置、数据导出
- 第一版可以没有：
  1. 数据看板（热力图、红黄绿灯预警）
  2. 一户一档的完整报告自动生成
  3. 业主扫码查看功能
  4. 短信通知（先用小程序内通知/模板消息）
- 后续再做：
  1. 数据看板模块（宏观进度、问题热力图、整改进度预警）
  2. 一户一档报告自动生成导出（按附件1-5格式）
  3. 业主端扫码查看
  4. 离线暂存+联网自动上传
  5. 短信通知提醒

## 不做什么

- 第一阶段明确不做：
  - 数据看板的图表和热力图
  - 业主端功能
  - 复杂的统计分析
  - 与第三方施工管理系统的对接
- 用户已经否定的方向：无
- 不能悄悄保留的旧概念：无（新项目）
- 需要用户确认的兼容取舍：附件表格导出格式需严格按照需求文档中的观感表、实测表、公区表格式。

## 关键风险

- 登录/角色/权限：
  - 微信授权登录，需获取用户手机号或 openid
  - 4 种角色（管理员、查验员、整改员、业主），需后端做接口权限校验
  - 数据权限：不同角色只能看到与自己责任区域相关的问题
- 支付/会员/额度：不涉及
- 上传/文件/隐私：
  - 问题照片强制上传，涉及图片存储（需评估云存储方案）
  - 房屋质量数据涉及未来业主隐私，需数据加密和权限控制
- 第三方 API/SDK：
  - 微信小程序 SDK（扫码、拍照、用户授权）
  - 二维码生成服务
- 本地系统/桌面软件/自动化能力：管理后台为 Web 端，需单独部署
- 生产环境/真实用户：
  - 微信小程序需通过官方审核上线
  - 需域名备案、HTTPS、ICP 许可证等合规事项
  - 安卓和 iOS 均需测试

## 验收标准

- 用户如何确认第一版可用：
  1. 管理员能导入楼栋数据，生成二维码
  2. 查验员扫码能进入对应户的问题上报页
  3. 能选择问题类型、拍照、提交问题
  4. 整改员能收到任务、上传整改证明
  5. 查验员能复查确认、问题闭环
  6. 管理员能在后台导出数据
- AI 需要提供的技术证据：
  - 小程序各页面截图（正常流程 + 空状态 + 错误状态）
  - API 请求/响应日志
  - 真机测试截图（安卓 + iOS）
- 需要截图或用户点击确认的地方：核心闭环全流程走通
- 不能声称完成的条件：未在真机上测试、未走通完整闭环、未做权限校验

## 停止条件

- 本阶段完成条件：核心闭环链路可完整走通 + 管理后台可做基础管理
- 进入下一阶段前需要的证据：真机测试截图、API 合同验证通过、导出功能可用
- 当前未验证项：后端技术栈、微信小程序认证状态、云存储方案
