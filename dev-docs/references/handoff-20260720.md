## 一、项目概要

住宅工程质量分户验收微信小程序。使用 uni-app (Vue 3) + Vite 构建，编译到微信小程序平台。

**角色**：查验员（上报+复查）、整改员（整改）、监理（复查确认）、管理员（后台管理）

**核心闭环**：逐项检查（观感/实测/公区）→ 有问题则指定整改人 → 整改人接收 → 提交整改结果 → 复查确认 → 闭环 / 退回继续整改

## 四、关键设计决策

| 决策 | 方案 | 原因 |
|------|------|------|
| 整改人选择 | 底部弹窗 + 搜索 | 微信小程序下拉列表不稳定，弹窗体验更好 |
| 问题预设来源 | `inspection-forms.js` 定义结构 → API `getIssuePresets(type)` 返回预设值 | 方便后端管理后台动态配置预设 |
| 无问题验收 | 检查页逐项"合格/不合格"→ 提交时自动标记 TAB 状态 → 三 TAB 全合格才算无问题 | 不能跳过检查直接点"无问题"，需要通过检查驱动 |
| 检查状态流 | `pending → passed/has_issues`，存于 `household.inspectionStatus` | Mock 层持久，从子页面返回自动刷新 |
| Mock ↔ 真实后端切换 | `.env` 文件 `VITE_USE_MOCK=true/false`，编译期常量注入 | 不依赖运行时 import，避免 tree-shaking |
| API 传输层 | `request()` 包装器：自动注入 token、401 处理、网络异常回退 | 等后端就绪后直接可用 |
| 文件上传 | `uploadRequest()` 包装器：同样支持 token、401 | 图片/视频/语音上传统一入口 |
| 户型 ID 编码 | `h_${b}${u}${f}${r}` 无分隔符（如 h_1010 = 1#楼西单元1层101室） | 历史设计，存在歧义风险但当前可用 |
| 角色门禁 | 前端 store.user.role 控制 UI | 最终权限需后端校验 |
| uni-app 版本 | npm alpha 版 | Vue 3 的 npm 渠道只有 alpha，正式版只在 HBuilderX |

## 五、页面流转

```
Tab 首页 ──→ 问题库 / 待办跳转
Tab 扫码 ──→ 扫码 → 户操作中心
                ├── 问题上报 → inspect.vue（观感/实测/公区）
                ├── 批量上报 → batch-report.vue
                ├── 二维码 → qrcode.vue
                └── 检查进度 → 三个 TAB 状态卡片（点击跳 inspect）
                  ├── 全部合格 → 验收完成
                  ├── 有问题 → 整改流程
                  └── 全部闭环 → 整户审核确认
                │
                └── 问题列表 → issue-detail.vue
                      ├── 接收整改（pending → rectifying）
                      ├── 提交整改（rectifying → pending_review）
                      └── 复查确认（pending_review → closed / rectifying）

Tab 任务 ──→ 任务列表（待整改/整改中/待复查/已闭环）
Tab 我的 ──→ 个人中心 + 通知 + 切换角色
```

## 六、状态体系

### 问题流转

```
pending（待整改，已指定整改人）
  → rectifying（整改中，整改人点击接收后）
    → pending_review（待复查，整改人提交结果后）
      → closed（已闭环，复查通过）或 rectifying（退回整改）
```

### 检查状态（inspectionStatus，存于每户）

| 字段 | 可能值 | 含义 |
|------|--------|------|
| `visual` | pending / passed / has_issues | 观感检查 |
| `measure` | pending / passed / has_issues | 实测检查 |
| `public` | pending / passed / has_issues | 公区检查 |

三 TAB 全 `passed` → 户验收自动变为 `已完成（无问题）`

## 九、技术债

1. **dev 构建产物被微信开发者工具锁定**：修改源码后热重载失效，必须关工具→重建→重开
2. **户型 ID 无分隔符**：`h_${b}${u}${f}${r}`（如 `h_1010`），楼栋号≥10 时与低栋号可能混淆
3. **输入框**：微信小程序原生 `<input>` 不兼容 `box-sizing`，禁止垂直 padding
4. **uView Plus**：大量 Sass `@import` deprecated warning
5. **颜色硬编码**：部分 scoped 样式中仍有直接颜色值
6. **uni-app alpha 版**：npm 渠道 Vue 3 只有 alpha，热重载不稳定
7. **Mock 代码随生产包发布**：`import * as mock from '@/mock'` 是静态导入，即使 `USE_MOCK=false` 时不会执行 Mock 函数，但 import 语句仍在 bundle 中。应改为动态 `if (USE_MOCK) { const mock = await import('@/mock') }`
8. **API 路径未与合同对齐**：约 10 处路径不匹配（见审计章节），联调前必须对齐
9. **硬编码楼层数 1-12**：`household-center.vue` 和 `inspect.vue` 中楼层写死 12 层，应从后端配置获取
10. **检查表单配置硬编码**：`config/inspection-forms.js` 中 42 项检查项定义应来自后端 API，而非前端硬编码

## 十五、需求审计发现（2026-07-20）

### 1. API 路径与合同不一致

`api-contract.md` vs `api/index.js` 中的真实请求路径：

| 合同路径 | 代码中路径 | 问题 |
|---------|-----------|------|
| `POST /api/auth/login` | `/api/login` | 路径前缀不符 |
| `GET /api/auth/profile` | `/api/user/profile` | 路径不符 |
| `GET /api/units/:unitId/floors` | **缺失** | API 未实现（当前硬编码 1-12 层） |
| `GET /api/floors/:floorId/households` | `/api/households` | 路径和参数不符 |
| `POST /api/admin/buildings/import` | **缺失** | 管理后台 API 未实现 |
| `GET /api/rectify/tasks` | `/api/issues/rectify-tasks` | 路径不符 |
| `PUT /api/issues/:issueId/rectify/start` | `/api/issues/xxx/rectify` | 方法和路径不符 |
| `GET /api/review/pending` | `/api/issues/pending-reviews` | 路径不符 |
| `GET /api/admin/statistics` | `/api/statistics` | 路径不符 |
| 管理后台 6.1-6.5 | **缺失** | 5 个接口未实现 |
| `getFormItems` / `getAllIssues` / `getEstimatedValues` / `checkOverdue` / `searchRectifiers` | 代码中有但合同未定义 | ⚠️ 需确认这些是最终接口还是临时 Mock |

### 2. Mock 模式问题

| # | 问题 | 影响 |
|---|------|------|
| 1 | **Mock 代码随生产包发布**：`import * as mock` 是静态导入，Vite tree-shaking 无法移除有副作用的模块 | 违反 AGENTS.md 原则 |
| 2 | **`wx.login()` 未调用**：Mock 模式下直接注入固定用户"张查验" | 无法验证微信登录流程 |
| 3 | **图片/视频为本地临时路径**：inspect.vue 拍照后存 `tempFilePath`，未压缩上传 | 真实场景需压缩 → 上传 OSS → 回传 URL |
| 4 | **电子签名为 base64 字符串**：Mock 中直接存，真实场景需上传后传 ID | 需与后端对齐签名方案 |
| 5 | **角色数据无区域过滤**：Mock 未按 `buildingIds` 过滤数据 | 无法验证数据权限 |

| 需求 | 现状 | 差距 |
|------|------|------|
| 查验员仅可复查**自己负责区域**的问题 | 所有 inspector/supervisor 角色均可复查 | 缺少区域粒度 |
| 整改员**按指派接收**任务 | Mock 按 rectifier===nickname 匹配 | 联调时需按 rectifierId |
| 数据权限按**责任区域**过滤 | 未按 buildingIds 过滤 | 缺少区域数据权限 |
| 二维码需**稳定生成+防伪水印** | Canvas 模拟生成 | 不可上线 |
| 离线→在线**自动上传** | 有 enqueue 无 auto flush | 缺少自动重试逻辑 |
| 问题库**按房间号过滤** | 只有楼栋级筛选 | 缺少房间粒度 |
| 无问题标记为**每项勾选** | 逐项判定合格/不合格 | 需确认客户是否接受 |
