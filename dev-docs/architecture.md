# Architecture Truth — 分户验收小程序

## 推荐架构

- 推荐产品形态：微信小程序（前端）+ Web 管理后台 + 后端 API 服务
- 推荐技术路线：
  - **小程序端**：Uni-app (Vue 3) + uView Plus 3.x UI 组件库，使用 HBuilderX 或 Vue CLI 开发
  - **管理后台**：单页面 Web 应用（技术栈待定，建议 Vue 3 + Element Plus 或 React + Ant Design）
  - **后端 API**：由后端老师独立负责，技术栈由其决定
  - **API 合同**：RESTful JSON API，合同文档在 `dev-docs/api-contract.md`
- 推荐部署形态：
  - 小程序端：微信审核上线
  - 后端 API：云服务器部署（由后端老师负责）
  - 管理后台：静态资源部署到同一服务器或 CDN
  - 图片/文件存储：云存储（OSS/COS，方案待确认）
- 为什么这是主路线：
  - Uni-app (Vue 3) 成熟稳定，Vue 语法比微信原生 WXML 更易维护
  - uView Plus 提供丰富的 UI 组件，保证界面美观统一，减少重复造轮子
  - 前后端分离，并行开发，通过 API 合同解耦
  - 管理后台独立部署，不干扰小程序
  - 如需扩展 H5 或 App，无需重写代码
- 被拒绝路线和原因：
  - 微信原生开发：语法古老，UI 组件需从零写，开发效率不如 Uni-app，界面美观度难保证
  - Taro (React)：React 在小程序生态不如 Vue 成熟，组件库选择少
  - 小程序云开发：不适合客户交付场景，数据可控性不如自建后端

## 责任边界（重要）

```
┌─────────────────────────────────────────────────────┐
│  本项目的责任范围（你负责的部分）                      │
│                                                      │
│  ┌──────────────┐    ┌──────────────────────┐        │
│  │  微信小程序   │    │  Web 管理后台         │        │
│  │  (前端)      │    │  (前端)               │        │
│  └──────┬───────┘    └──────────┬───────────┘        │
│         │                       │                     │
│         └───────────┬───────────┘                     │
│                     │                                │
│              调用后端 API                             │
│                     │                                │
├─────────────────────┼────────────────────────────────┤
│                     │                                │
│              ┌──────┴───────┐                        │
│              │  后端 API     │  ← 后端老师负责        │
│              │  (不归你管)   │                        │
│              └──────────────┘                        │
└─────────────────────────────────────────────────────┘
```

- 你负责：小程序所有页面、组件、交互逻辑、API 调用、管理后台前端
- 后端老师负责：API 接口实现、数据库设计、权限校验、数据存储
- 你需要和后端老师对齐的：API 合同（接口路径、参数、返回值、错误码）

## Owner Map

| 概念 | Owner | 不能由谁拥有 | 验证方式 |
| --- | --- | --- | --- |
| 产品边界 | dev-docs/project-brief.md | 代码注释 | 对照需求文档 |

| 数据库/schema | 后端老师 | 小程序端 | 后端迁移脚本 |
| 登录/权限 | 后端 API 校验 | 小程序路由 | 未登录/无权限时后端返回 401/403 |

## 调用链

```text
用户扫码/打开小程序
  -> 微信授权登录（wx.login + 后端 token）
  -> 小程序页面（WXML/WXSS/JS）
  -> 调用 wx.request / wx.uploadFile
  -> 后端 API（鉴权 → 业务处理 → 数据库读写）
  -> 返回 JSON 响应
  -> 小程序更新页面状态
  -> 用户看到结果

管理后台
  -> 浏览器打开 Web 页面
  -> 管理员登录
  -> 调用后端 API
  -> 数据管理、导出
```

## 小程序端架构

- 框架：**Uni-app (Vue 3) + uView Plus 3.x**
  - 主目标平台：微信小程序（上线）
  - 可扩展平台：H5（管理后台轻量版）、App（未来需要）
  - 跨平台通过 uni-app 条件编译 `#ifdef MP-WEIXIN` / `#ifndef` 处理差异
- 设计风格：**美观简洁**——现代工具型风格，移动端适配，适合工地现场使用
  - 大按钮、清晰文字、高对比度（户外/室内都看得清）
  - 充足的触控区域（最小 44px 点击区域）
  - 操作路径短、一步能完成的不用两步
- 全局样式：`uni.scss` 定义设计 token（颜色、字号、间距、圆角）
- 设计 token 文件：`styles/tokens.scss`
- 离线能力：因工地网络复杂，关键操作需支持离线暂存，网络恢复后自动上传
- 页面结构（Vue SFC）：
  ```
  pages/
    index/          # 首页/工作台（显示统计数据、待办任务、快速入口）
    scan/           # 扫码页面（调用 wx.scanCode）
    scan-result/    # 扫码结果页（显示户信息，进入问题上报）
    report/         # 问题上报（观感/实测/公区选择、拍照、描述、期限）
    batch-report/   # 批量上报（多户复选同一问题）
    issue-detail/   # 问题详情（全流程记录）
    rectify/        # 整改操作
    review/         # 复查确认
    household/      # 一户一档（该户所有问题看板）
    mine/           # 我的/个人中心
  ```
- 组件结构（.vue 单文件组件）：
  ```
  components/
    StatusTag.vue       # 状态标签（待整改/整改中/待复查/已闭环）
    IssueCard.vue       # 问题卡片
    PhotoUploader.vue   # 拍照上传组件（强制至少1张，最多9张）
    BuildingPicker.vue  # 楼栋-单元-层-户级联选择
    HouseholdPicker.vue # 多户选择组件（批量上报用）
    BatchReportItem.vue # 批量上报条目
    NoIssueCheck.vue    # 无问题勾选组件
    EmptyState.vue      # 空状态占位
    LoadingSpinner.vue  # 加载状态
  ```
- UI 原则：
  - 每个页面处理 5 种状态：正常态、加载态、空状态、错误态、网络异常态
  - 操作反馈即时：按钮点击有 loading、成功有 toast、失败有明确提示
  - 复用优先：同一 UI 模式跨页面统一组件
  - 颜色语义一致：绿=成功/闭环、黄=待处理/超时、蓝=进行中、红=错误/不合格
  - uView Plus 组件优先：按钮、表单、弹窗、Tab、列表等基础组件使用 uView Plus，避免重复造轮子
- 禁止混用的样式方式：禁止内联 style 覆盖全局 token；禁止硬编码颜色值（必须用 token 变量）。

## 管理后台架构

- 框架：待后端老师确定技术栈后配合选择（Vue 3 或 React）
- 核心页面：
  - 楼栋数据管理（导入、查看、编辑）
  - 人员管理（角色分配、权限配置）
  - 问题列表查看
  - 数据导出（Excel/Word，按附件格式）
- 设计风格：后台管理型，专业简洁

## 后端架构（由后端老师决定，以下为建议）

- 后端是否需要：是
- 后端负责什么：所有业务逻辑、权限校验、数据存储、文件处理、通知推送
- API 合同位置：`dev-docs/api-contract.md`
- 请求生命周期：鉴权中间件 → 参数校验 → 业务处理 → 数据库操作 → 响应
- 统一响应格式：
  ```json
  {
    "code": 0,
    "message": "success",
    "data": { ... }
  }
  ```
- 权限和安全：角色权限由后端校验，小程序端不做权限判断

## 第三方和外部能力

- 微信小程序 SDK：扫码（wx.scanCode）、拍照/相册（wx.chooseImage）、用户授权（wx.login、wx.getUserProfile）
- 二维码生成：后端提供，需包含楼号+单元号+房号信息
- 图片存储：云存储方案待确认（OSS/COS/其他）
- 未验证项：微信小程序认证账号、云存储选型

## 开发策略：Mock 先行

由于你不负责后端开发，采用 **Mock 驱动的前端先行**策略：

1. **Mock 数据层** — 在 `mock/` 目录中实现模拟数据服务，返回与 API 合同一致的 JSON
2. **API 抽象层** — 在 `api/` 目录中封装请求方法，开发时指向 mock 服务，联调时切换到真实后端
3. **开发顺序**：
   ```
   Mock 数据定义 → 前端页面/组件实现 → 页面与 Mock 联调
     → 后端 API 就绪 → 切换到真实后端联调
   ```
4. **Mock 覆盖范围**：所有 API 合同中的接口都需要 Mock 实现，包括正常数据、空数据、错误响应
5. **切换方式**：通过环境变量或配置文件一键切换 mock/真实后端
6. **底线**：Mock 数据仅用于前端开发调试，**绝不可随上线包发布**

## 禁止路径

- 禁止小程序端做权限判断（仅做 UI 展示控制，真正权限由后端校验）
- 禁止小程序端写业务逻辑（状态机、数据校验规则均由后端控制）
- 禁止硬编码接口地址（统一管理在 config 中）
- 禁止在页面中直接操作数据（通过 API 层统一调用）
- 禁止使用 mock 数据上线（开发阶段允许 mock 用于调试）

## 验证方式

- 本地开发：HBuilderX 打开项目 → 运行到微信开发者工具
- 构建命令：HBuilderX → 发行 → 微信小程序，或 `npm run build:mp-weixin`
- 真机测试：微信开发者工具预览扫码 + 安卓/iOS 各一部
- 跨平台检查：条件编译部分需验证在目标平台行为正确
- API 联调：与后端老师约定联调环境
- Git checkpoint：每个子阶段完成后提交
