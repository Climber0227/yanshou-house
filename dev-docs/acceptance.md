# Acceptance Truth — 分户验收小程序

## 验收门禁

- 验收必须匹配改动风险，不能只跑最轻命令就宣布完成。
- 声称"完成""修好""通过"前，必须提供本轮实际运行过的证据。
- 如果无法运行某项关键验收，必须说明原因、影响范围和可替代证据。
- 发现失败时，先复现、收集证据、建立可证伪假设，再修改。
- warning、lint warning、TODO 都按缺陷处理；除非明确记录为非本轮债务，否则不能带着它们声称完成。

**第一阶段门禁**：核心闭环链路可完整走通 + 管理后台可做基础管理。

### 各子阶段门禁

**子阶段 1：项目骨架搭建**
- [ ] uni-app 项目可在 HBuilderX 中正常打开
- [ ] 全局样式 token 定义完成（uni.scss / styles/tokens.scss）
- [ ] 所有页面目录和基础路由创建完成
- [ ] uView Plus 组件库集成完毕
- [ ] API 请求层封装完成（基于 uni.request，统一错误处理、token 管理）
- [ ] Mock 数据层完成覆盖所有 API 接口（正常数据 + 空数据 + 错误响应）
- [ ] 环境变量/配置可一键切换 mock ↔ 真实后端
- [ ] Mock 模式下各页面可独立预览
- 阻断条件：项目无法在 HBuilderX 或微信开发者工具中打开

**子阶段 2：户码管理与扫码**
- [ ] 楼栋数据能正确展示（用 mock 数据）
- [ ] 二维码能正常生成和展示（二维码下方显示楼号-单元-房号）
- [ ] 微信扫码能正确解析并跳转到对应户的问题上报页
- [ ] 楼栋/单元/层/户级联选择可用
- 阻断条件：扫码流程不通

**子阶段 3：问题上报**
- [ ] 观感、实测、公区三类验收可选择
- [ ] 问题描述预设列表可用、支持自定义输入
- [ ] 拍照上传功能可用（调用相机 + 相册选择）
- [ ] 视频录制和语音备注可选
- [ ] 提交成功后状态更新、提示明确
- [ ] 必填项校验、空状态、网络错误提示
- 阻断条件：拍照上传不通

**子阶段 4：整改流程**
- [ ] 整改员能看到分配给自己的任务列表
- [ ] 能上传整改照片、填写整改措施
- [ ] 状态正确流转：待整改→整改中→待复查
- [ ] 查验员能看到"待复查"的问题列表
- [ ] 复查确认功能可用（合格/不合格）
- [ ] 合格后状态变为"已闭环"
- [ ] 不合格退回整改，状态回到"整改中"
- [ ] 整改时限超时提醒（如有）
- 阻断条件：状态流转不完整

**子阶段 5：管理后台 + 导出**
- [ ] 管理员可查看、导入、编辑楼栋数据
- [ ] 人员管理和角色分配可用
- [ ] 数据可按附件格式导出（观感表/实测表/公区表）
- [ ] 导出文件格式与需求文档附件一致
- 阻断条件：导出格式与附件不一致

**子阶段 6：集成联调 + 真机测试**
- [ ] 与后端 API 联调通过（所有接口调用正常）
- [ ] 安卓真机测试通过
- [ ] iOS 真机测试通过
- [ ] 弱网环境测试（至少 3G 网络下可用）
- [ ] 完整闭环走通至少 3 户
- 阻断条件：任一平台真机测试未通过

## 证据记录

每个子阶段完成后，填写证据：

```
子阶段：____
完成日期：____
通过的验收项：
  - [x] 项 1 — 截图/日志路径
  - [x] 项 2 — 截图/日志路径
失败的验收项：
  - [ ] 项 X — 失败原因、处理方式
未覆盖的风险：
  - 风险描述、为什么没覆盖
下一子阶段准备：
  - 是否满足进入条件
```

证据类型要求：
- 页面功能：HBuilderX 预览 / 微信开发者工具截图 + 真机截图
- API 联调：请求/响应日志或抓包截图
- 导出功能：导出文件截图（与附件格式对比）
- 状态流转：各状态节点截图

核心闭环链路可完整走通 + 管理后台可做基础管理。

具体检查：

### 进入下一阶段前的证据

- 真机测试截图/录屏（核心闭环全流程）
- API 合同验证通过（所有接口联调成功）
- 导出功能可用（导出文件与附件格式对比截图）
- 未闭合风险和未验证项清单

## 漂移检查

### 功能漂移预警

以下情况视为需求漂移，必须先更新 `project-brief.md` 再继续开发：
- 新增第一阶段范围外的模块（如数据看板、业主端）
- 修改核心流程（如增加审批环节、改状态机）
- 新增角色或权限规则
- 修改验收表格格式（偏离附件模板）

### 架构漂移预警

以下情况视为架构漂移，必须先更新 `architecture.md` 再继续开发：
- 在小程序端实现业务逻辑或权限判断
- 新增第三方 SDK 或依赖
- 修改 API 合同但不通知后端老师
- 在页面中直接硬编码接口地址或配置

### 质量漂移预警

以下情况视为质量漂移，必须停止当前工作并修复：
- 已有功能回归（之前通过的门禁现在失败）
- 同一 UI 组件在多个页面重复实现
- 全局样式被内联样式覆盖
- 出现未被记录的 mock 数据或硬编码

### 发现漂移时的处理

1. 停止当前开发
2. 记录漂移证据（哪个文件、什么变化、为什么偏离）
3. 更新对应真源文档
4. 与用户确认是否接受新方向
5. 确认后再继续开发
